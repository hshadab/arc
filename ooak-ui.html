How This Demo Works

This single‑file UI calls a local API to run a proof‑gated approval flow. It can generate an off‑chain zkML proof (JOLT‑Atlas) and an on‑chain‑verifiable Groth16 “proof‑of‑proof” that binds the model’s decision.

  • Inputs: Provide amount and risk. The server derives a decision and confidence.
  • zkML (JOLT‑Atlas): If available, the server runs the real prover and returns a commitment proofHash to the zkML proof bytes.
  • Groth16 proof‑of‑proof: Proves [decision, confidence] (and optionally proofHash) so a verifier contract can check them.
  • On‑chain verify: Read‑only contract call returns true/false for the Groth16 proof.
  • Approval: The JSON you see is the “approval envelope” you would pass to a secure action.

Relation to Circle OOAK

Circle’s OOAK uses @secure_tool and a WorkflowManager to approve intents before tools execute. Here, we make approval cryptographic:

  • Gate: A secure tool executes only with a valid approval envelope.
  • Verify: Manager verifies zkML (off‑chain) and Groth16 (on‑chain) before issuing a workflow id.
  • Pin: Enforce specific model versions and input commitments for auditability and privacy.
  • Audit: On‑chain verification and proof hashes create a portable, tamper‑evident record.
